# Node routing model (v4.3+)

How cortex's mining-layer agents decide which node an extracted item belongs to.

## Inputs each miner receives

- **`node_inventory`** — list of all node IDs and their inferred type, generated by scanning `<config-root>/memory/`. Types: `project`, `domain`, `person`, `user`, `archive`. Inferred from front-matter `type:` field, directory location, or filename convention.
- **`node_summaries`** — for each node:
  - The node's Scope section (if present — see below)
  - First ~500 chars of the Summary section
  Enough to route without loading every node fully.
- **`dashboard_snapshot`** — current `DASHBOARD.md` content. Tells the miner what's actively tracked vs. dormant.

## Scope section convention (domain nodes)

Every domain node should have a Scope section as its first body section (above Summary). Adopted via the one-time migration in `/end-day` Pre-chain B. Schema:

```markdown
## Scope
- **Topics:** prospecting, outreach, ICP, signal sources, CRM hygiene
- **Aliases:** bd, business-dev, sales
- **What goes here:** anything cross-cutting about how I find, qualify, and contact prospects
- **What does NOT go here:** specific client engagement state (use `client:<name>`), referral-ask drafting (lives in `referral-engine` plugin data)
```

Four fields:

| Field | Purpose |
|---|---|
| **Topics** | Hard keyword match in router step 2 |
| **Aliases** | Match abbreviations / synonyms the user says |
| **What goes here** | Semantic confirmation when topic match is fuzzy |
| **What does NOT go here** | Explicit exclusions with redirect hints |

The "what does NOT go here" field is the most useful in practice — it prevents leak across adjacent domains.

## Routing algorithm (each extracted item)

Apply in order. Stop at the first match.

1. **Hard match** — if the item explicitly names a node by ID or by an alias from the node's Scope section, route there.
2. **Topic match** — if the item's keywords overlap a domain node's `Topics:` list ≥ 2 keywords OR semantically match the `What goes here:` prose, route there. Honor `What does NOT go here:` exclusions strictly.
3. **Person match** — if the item is primarily about a specific person and not anchored to a project, route to `person/<slug>.md` if it exists. If no page exists, this is a graduation signal — propose with `target_node: person:<slug>` and `node_type: new`.
4. **User match** — preferences, corrections, patterns about the user themselves → `user` node.
5. **Cross-cutting** — if the item legitimately belongs to two nodes (e.g., "Acme's procurement gotcha is the 3rd time we've seen this; generalize to bizdev"), produce TWO proposals linked via `cross_ref`.
6. **No fit** — if confidence on all of the above is < medium, propose a new node with a suggested ID, type, and one-line scope (`node_type: new`).

Casual mentions and weak signals fall through to step 6's "no fit" branch — the review gate makes new-node creation explicit so the taxonomy doesn't bloat silently.

## Routing edge cases

- **Scope section missing on a candidate domain node** → fall back to topic match against the node's Summary prose. Log in confidence notes ("Routed by Summary-prose match — Scope section absent, recommend running Scope migration"). Lower routing accuracy is the cost of skipping the migration.
- **Two nodes both match topic-wise** → pick the one with stronger keyword overlap; flag the other as cross-ref candidate; if overlap is identical, prefer the more-specific node (project node beats domain node) and surface ambiguity in confidence notes.
- **Item references a contact who has no person page** → if the item is primarily relationship context, propose `new_node_suggestion` for a person page. The review gate decides whether to create.
- **Item is a correction that contradicts an existing entry** → use `update_type: correction` with the existing entry quoted in `rationale`. The review gate makes the supersede decision explicit (don't silently overwrite).

## What the router never does

- Auto-create nodes silently — all new nodes go through the review gate's explicit confirmation
- Route to archived nodes — archived means user already decided this thread is closed
- Route to `user.md` for project-specific knowledge — `user.md` is for preferences/patterns about the user themselves, not project state
- Punt — every proposal must have a `target_node` (existing or `new_node_suggestion`). If the router can't decide, it picks the best fit with `confidence: low` and a rationale, not "I don't know"
